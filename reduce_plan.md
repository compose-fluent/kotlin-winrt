# Generated Projection Size Reduction Plan

## Baseline

- [x] Windows SDK prebuilt projection jar is about 53.9 MB with 24,839 `.class` files.
- [x] Windows SDK compressed class bytes are about 48.6 MB.
- [x] Windows App SDK prebuilt projection jar is about 26.4 MB with 12,502 `.class` files.
- [x] Windows App SDK compressed class bytes are about 23.7 MB.
- [x] TSV entries in the published jars are effectively zero-size and are not the optimization target.
- [x] Most size comes from repeated generated class templates, especially per-type metadata/support output rather than a small number of oversized resources.
- [x] For the next comparison pass, fetch the unoptimized projection jar baseline from the published Maven Central/Sonatype snapshot instead of rebuilding `master` locally, then compare the locally optimized branch output against that immutable published artifact.

## Goal

- [x] Reduce repeated generated `.class` output while preserving existing Maven coordinates.
- [x] Preserve projected API compatibility for `winrt-projections-windows-sdk` and `winrt-projections-windows-app-sdk`.
- [x] Preserve runtime hot-path performance: no runtime TSV parsing, classpath scanning, or reflection-based projection discovery.
- [x] Preserve multi-module classpath safety when several WinRT projection-owning jars are consumed by the same application.
- [x] Keep artifact-specific generated support classes owner-qualified instead of reintroducing fixed global registry class names.

## Implementation Queue

- [x] Measure generated class categories and repeated templates before each optimization step.
- [x] Optimize per-type `Metadata` boilerplate first by moving repeated descriptor construction into shared helpers while keeping direct generated access and lazy caching.
- [x] Optimize repeated nested/support class templates second, including `NativeProjection` and event/helper scaffolding only where behavior remains `.cswinrt`-aligned.
- [x] Keep shared helpers stateless or semantically global; keep artifact-specific support state under owner-qualified names such as `windows_sdk`, `windows_app_sdk`, or a sanitized artifact identity suffix.
- [x] Add regression coverage that two independent WinRT projection modules can be compiled and consumed together without fixed-FQCN support class collisions.
- [x] Keep generated output deterministic and avoid broad source slimming unrelated to repeated generated bytecode.

## Validation Checklist

- [x] Run generator tests.
- [x] Run Gradle plugin tests.
- [x] Run compiler plugin tests.
- [x] Compile prebuilt projection modules.
- [x] Publish Windows SDK and Windows App SDK projection artifacts to Maven Local.
- [x] Compare before/after jar size, class count, class category totals, and compressed class bytes.
- [x] Run a multi-module projection consumption regression to verify classpath collision safety.
- [x] Run the WinUI sample compile/run path after any generator behavior change that affects WinAppSDK output.

## Constraints

- [x] Do not split artifacts in this slice.
- [x] Do not reduce the default projected namespace/type surface in this slice.
- [x] Do not introduce runtime lookup paths that trade jar size for slower projection calls.
- [x] Do not add fixed global generated registry/support FQCNs for module-specific state.
- [x] Keep `PLAN.md` as the repository-wide plan; this file is the focused working plan for generated-size reduction.

## Next Slice: Module-Local Platform ABI Calls

- [x] Measure repeated `WinRTProjectionIntrinsic.*` call shapes per generated projection artifact and group them by exact ABI argument carrier sequence, return handling, and vtable slot call form.
- [x] Replace broad per-call compiler-plugin FFM IR expansion with generated module-local ABI call owners such as owner-qualified `ModulePlatformAbiCall` support classes.
- [x] Generate only fixed-signature ABI call functions for observed module shapes, for example `call_Int32_Float(instance, slot, pa: Int, b: Float)`, and do not introduce `vararg`, `Array<Any?>`, `List<Any?>`, or generic boxed argument paths.
- [x] Keep support class names artifact-owner-qualified so Windows SDK, Windows App SDK, and application-generated projection modules cannot collide on the same classpath.
- [x] Prefer an existing runtime/platform ABI helper only when the runtime already exposes an exact fixed-shape path; do not add new runtime surface or runtime registries for this size optimization.
- [x] Model the generated support as `expect` / `actual`: common generated projection code calls the module-local expect API, JVM actuals implement the fixed-shape calls with FFM, and later native actuals can be generated or implemented without changing projected public APIs.
- [x] Decide whether JVM actual FFM bodies are emitted directly by `winrt-generator` or produced by the compiler plugin from generated shape declarations; choose the path with smaller class output, stable KMP source-set behavior, and no runtime hot-path regression.
- [x] Retire `WinRTProjectionIntrinsic` call emission for covered generated projection shapes only after module-local ABI call generation and compiler/plugin validation are in place.
- [x] Validate generated bytecode reduction against Windows SDK and Windows App SDK jars, including compressed class bytes, uncompressed class bytes, class count, and representative method bytecode.
- [x] Keep the before/after comparison script able to consume a Maven-resolved baseline jar path and a local optimized jar path so size analysis does not require a local baseline compilation.
- [x] Validate performance with fixed-shape ABI calls against the current direct intrinsic lowering so the optimization does not trade class size for boxing, megamorphic dispatch, or slower hot-path calls.

## Follow-Up Size Analysis

- [x] Re-measure the optimized jars by class category before the next implementation slice: Windows SDK still has about 66.27 MiB uncompressed top-level type bytecode and 37.64 MiB uncompressed `$Metadata` bytecode; Windows App SDK still has about 29.16 MiB uncompressed top-level type bytecode and 17.09 MiB uncompressed `$Metadata` bytecode.
- [x] Treat the remaining direct `WinRTProjectionIntrinsic.*` sites as the next primary bytecode-reduction target: optimized Windows SDK generated source still has about 16,148 direct intrinsic sites, and optimized Windows App SDK generated source still has about 8,527 direct intrinsic sites.
- [x] Prioritize static, factory, struct-return, object-return, projected-interface-return, and projected-runtime-class-return call shapes before another scalar getter/setter pass, because samples such as `Launcher$Metadata` and `Colors$Metadata` still inline large FFM, scope, readback, and cleanup blocks per member.
- [x] Extend module-local fixed-shape ABI support to static and factory methods so large `$Metadata` methods can delegate to owner-qualified support functions while preserving public companion/metadata API entry points.
- [x] Extend module-local fixed-shape ABI support to struct return and struct setter paths by moving repeated native allocation, vtable call, `fromAbi`, `copyTo`, `disposeAbi`, and close/finally scaffolding into typed support functions where the struct adapter and ABI shape are fixed.
- [x] Extend module-local fixed-shape ABI support to nullable and non-null projected runtime-class/interface return paths by moving repeated pointer-slot readback, null checks, `RawComPtr` wrapping, and projected wrapper construction into typed owner-qualified helpers.
- [x] Analyze high-interface-count runtime classes such as `PackageManager` for repeated lazy interface-cache fields and getters, and consider replacing per-interface lazy delegate templates with a compact per-instance interface cache only if first-call and hot-call behavior stay at least as efficient as the current lazy fields.
- [x] Keep `NativeProjection` template reduction lower priority for the next pass because optimized Windows SDK `NativeProjection` bytecode is about 3.70 MiB uncompressed and optimized Windows App SDK `NativeProjection` bytecode is about 1.71 MiB uncompressed, much smaller than top-level type and `$Metadata` bytecode.
- [x] Do not introduce dynamic reflection, classpath scanning, vararg/object-array dispatch, or runtime registries for the follow-up pass; any new helpers must remain fixed-shape, owner-qualified, and compatible with multiple WinRT projection jars on one classpath.

## Current Direct Intrinsic Reduction Slice

- [x] Reject no-argument `Unit` / generated `NativeProjection.close()` routing through module-local `callUnit_Void`: it reduced source call sites but increased measured projection jar size because the added support function/class payload outweighed the small direct-call body savings.
- [x] Measure the rejected `callUnit_Void` experiment before reverting it: it covered 596 generated Windows SDK source sites and 372 generated Windows App SDK source sites, but the jar/class measurements moved in the wrong direction after clean rebuilds.
- [x] Reject the broad non-generic projected runtime/interface return helper experiment for inline result calls: it moved hundreds of call sites into module support but generated many return-type-specific support functions, growing measured jars instead of shrinking them. Do not continue this direction unless support functions can be grouped by ABI/readback shape without one helper per projected return type.
- [x] Measure module helper frequency before adding more helper families: current generated sources show many low-frequency helpers, with Windows SDK around 827 one-call helpers out of 1,283 and Windows App SDK around 287 one-call helpers out of 488. This means future size work should add a generation threshold or a pre-counted support-shape plan so one-off helpers fall back to direct intrinsic lowering instead of growing the module support class.
- [x] Regroup projected runtime-class/interface return helpers by raw ABI reference shape instead of projected return type: generated module helpers now use shared names such as `getNullableInspectable`, `getNullableUnknown`, `callInspectable_<shape>`, and `callUnknown_<shape>`, return `IInspectableReference` / `IUnknownReference`, and keep concrete `Metadata.wrap(...)` at the generated call site.
- [x] Keep this regrouping module-local and runtime-free: no runtime registry, no reflection/classpath scan, no `vararg` / object-array dispatch, and no fixed global support owner were added. The support helper still uses the existing intrinsic marker internally so compiler lowering emits one fixed-shape ABI body per module helper instead of inlining one body per projected return type.
- [x] Fix the WinUI sample host classpath collisions found during Windows App SDK `2.1.3` validation: generated event helper and generic ABI support file facades are now artifact-owner-qualified, so dependency projection jars and application-generated projection jars no longer define fixed `WinRTEventProjectionHelper_000Kt` / `WinRTGenericAbiSupportKt` classes with different compiler-plugin support initialization methods.
- [x] Add generator coverage for nullable runtime-class/interface getters and descriptor projected runtime-class returns to assert shared helper names, call-site wrapping, and absence of concrete projected `Metadata::wrap` references inside module ABI support.

- [x] Validate compiler-plugin baseline: `:winrt-compiler-plugin:test` passed after the regrouping, confirming the intrinsic-lowering path still compiles.
- [ ] Re-run full generator validation after local Windows.winmd parsing is fixed: full `:winrt-generator:test` currently still fails two real-WinMD parity tests because the installed `Windows.winmd` is reported as not a valid WinMD metadata file, unrelated to the projected-return helper regrouping.
