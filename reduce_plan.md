# Generated Projection Size Reduction Plan

## Baseline

- [x] Windows SDK prebuilt projection jar is about 53.9 MB with 24,839 `.class` files.
- [x] Windows SDK compressed class bytes are about 48.6 MB.
- [x] Windows App SDK prebuilt projection jar is about 26.4 MB with 12,502 `.class` files.
- [x] Windows App SDK compressed class bytes are about 23.7 MB.
- [x] TSV entries in the published jars are effectively zero-size and are not the optimization target.
- [x] Most size comes from repeated generated class templates, especially per-type metadata/support output rather than a small number of oversized resources.

## Goal

- [ ] Reduce repeated generated `.class` output while preserving existing Maven coordinates.
- [ ] Preserve projected API compatibility for `winrt-projections-windows-sdk` and `winrt-projections-windows-app-sdk`.
- [ ] Preserve runtime hot-path performance: no runtime TSV parsing, classpath scanning, or reflection-based projection discovery.
- [ ] Preserve multi-module classpath safety when several WinRT projection-owning jars are consumed by the same application.
- [ ] Keep artifact-specific generated support classes owner-qualified instead of reintroducing fixed global registry class names.

## Implementation Queue

- [ ] Measure generated class categories and repeated templates before each optimization step.
- [ ] Optimize per-type `Metadata` boilerplate first by moving repeated descriptor construction into shared helpers while keeping direct generated access and lazy caching.
- [ ] Optimize repeated nested/support class templates second, including `NativeProjection` and event/helper scaffolding only where behavior remains `.cswinrt`-aligned.
- [ ] Keep shared helpers stateless or semantically global; keep artifact-specific support state under owner-qualified names such as `windows_sdk`, `windows_app_sdk`, or a sanitized artifact identity suffix.
- [ ] Add regression coverage that two independent WinRT projection modules can be compiled and consumed together without fixed-FQCN support class collisions.
- [ ] Keep generated output deterministic and avoid broad source slimming unrelated to repeated generated bytecode.

## Validation Checklist

- [ ] Run generator tests.
- [ ] Run Gradle plugin tests.
- [ ] Run compiler plugin tests.
- [ ] Compile prebuilt projection modules.
- [ ] Publish Windows SDK and Windows App SDK projection artifacts to Maven Local.
- [ ] Compare before/after jar size, class count, class category totals, and compressed class bytes.
- [ ] Run a multi-module projection consumption regression to verify classpath collision safety.
- [ ] Run the WinUI sample compile/run path after any generator behavior change that affects WinAppSDK output.

## Constraints

- [x] Do not split artifacts in this slice.
- [x] Do not reduce the default projected namespace/type surface in this slice.
- [x] Do not introduce runtime lookup paths that trade jar size for slower projection calls.
- [x] Do not add fixed global generated registry/support FQCNs for module-specific state.
- [x] Keep `PLAN.md` as the repository-wide plan; this file is the focused working plan for generated-size reduction.
